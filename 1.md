## 1. 正确性

### 1.1 [应修] DELETE by alias 时 AgentHub reverse-sync 仍传原始 identifier

**位置**: `CubeAPI/src/handlers/templates.rsdelete_template` → `reverse_sync_agenthub_template`）

CubeMaster DELETE 已通过 `ResolveTemplateIdentifier` 解析 alias → `tpl-*`。但 CubeAPI 在删除成功后仍把 **path 上的原始字符串**（可能是 alias）传给：

```text

find_template_ids_by_template_or_source_snapshot(id)

```

AgentHub 表按 `template_id` / `source_snapshot_id` 精确匹配。结果：

- `DELETE /templates/my-app` 能删掉基础设施模板

- AgentHub 里挂着 `tpl-xxx` 的注册 **不会被 soft-delete**

- 留下悬空 AgentHub template

这是 fslongjin 2026-07-18 已指出、后续 commit 未覆盖的问题。只要环境启用了 AgentHub，且用户按文档用 alias 删除，就会踩中。

**建议**: 删除前先 resolve 出 canonical template ID（或让 delete API 返回 resolved ID）`reverse_sync` 必须用 canonical ID。

---

### 1.2 [应修 / 中] `claimTemplateAlias` 失败被吞掉，job 仍报 READY

**位置**: `image_job_runner.go` / `redo.go`

```go

if claimErr := claimTemplateAlias(...); claimErr != nil {

    // 仅 Infof / Warnf，不 fail job

}

```

影响：

- 调用方请求了 alias，build 显示成功，但 alias **可能根本没挂上**

- 并发抢同一 alias 时，失败方 “READY without alias”，调用方无感知

- 随后 `result_json` 里的 `info` 在 claim **之前**构造，即便 claim 成功`DisplayName` 也常是空的 → job 结果与 DB 不一致

**建议**:

1. claim 失败（非并发 duplicate）应 fail job，或至少把错误写进 `error_message` / 响应

2. claim 成功后 refresh `info.DisplayName` 再写 `result_json`

3. 并发 duplicate 若要 soft-success，需在 API/文档写清语义

---

### 1.3 [建议] PR 描述 “newest wins” 与实现不完全一致

实现保证的是：

- **唯一性**：同一时刻最多一个 template 拥有某 alias

- **顺序 rebuild**：后完成 READY 并成功 claim 的那个会抢走 alias（e2e 覆盖）

- **并发 claim**：DB 调度谁先 commit 谁赢，不是 “谁更新 / 谁更新时间更新”

可接受，但应改 PR/注释措辞，避免合入后被当成严格 “newest template wins” 语义。

---

### 1.4 [低] CubeAPI `is_valid_alias` 不拒绝 `tpl-snap-` 前缀

`get_template_by_alias` 入口会拒，但 create 路径 `alias_from_name("tpl-foo")` 仍会转发给 CubeMaster，再由 Go 侧 400。行为不统一，错误信息也更绕。

建议在 `is_valid_alias` 与 create 入口统一拒绝。

---

### 1.5 已确认正确的主路径（不再展开）

- READY 后 claim，失败 build 不抢旧 alias

- `GetTemplateByAlias` 走 `alias_key`，snapshot 不参与 alias 所有权

- sandbox create alias 解析仅在 v2 分支

- migration 双方言 dedup 均带 `kind='template'`

- DELETE/GET（CubeMaster）会 resolve alias

- e2e 覆盖 create/get/delete/validation/dedicated endpoint/rebuild 迁移

## 2. Regression

### 2.1 [应修 / 潜伏回归脚枪] `createDefinitionTx` 仍保留“创建时释放并占用 alias”

**位置**: `snapshot_ops.go` `createDefinitionTx`

当前 image/redo 调用：

```go

ensureTemplateDefinitionWithOptions(..., definitionCreateOptions{}) // DisplayName 为空

```

因此创建时 **不会**抢 alias——这是正确的。

但 `createDefinitionTx` 里 template-kind 分支仍然是：

- 若 `opts.DisplayName != ""` → 立即 release 旧 alias + INSERT 时带上 DisplayName（此时 status 仍是 `PENDING`）

也就是说：**旧的 “未 READY 就抢 alias” 行为仍以 API 形式存在**。任何未来调用方（或漏改路径）只要在 create 时塞 `DisplayName`，就会重新引入原先 blocking bug。

同时 `createDefinitionWithOptions` / `ensureTemplateDefinitionWithOptions` 注释仍写 “threading alias metadata (DisplayName)”，与 image/redo 现状矛盾，容易误导后续改动。

**建议**:

- 删除 template-kind 在 `createDefinitionTx` 里的 early release（alias 只允许经 `claimTemplateAlias`）

- 或对 template-kind **禁止** create 时带非空 DisplayName（assert / 返回错误）

- 更新过时注释

---

### 2.2 已排除的 regression

| 场景 | 结论 |

|------|------|

| 非 v2 legacy identifier 被当成 alias | 已修好，仅 v2 解析 |

| migration 误清 snapshot `display_name` | 已修好 |

| snapshot 与 template 抢同一 display_name | write/write 均隔离 |

| 删除中模板阻塞 alias 复用 | claim release 已去掉 `status <> DELETING` 过滤 |

### 2.3 [低] 升级存量数据

migration 会把既有 template 非空 `display_name` 纳入唯一约束并变成 alias。若历史上有人把 display_name 当自由标签用，升级后会变成全局唯一 alias。风险取决于存量数据；建议在 release note 提一句。

## 3. 代码冗余

| 项 | 说明 | 建议 |

|----|------|------|

| `createDefinitionTx` early alias 逻辑 | 对当前 image/redo 不可达，与 `claimTemplateAlias` 双轨 | 删除或禁止 create 时带 alias（见 2.1） |

| `ensureTemplateDefinitionWithOptions` 的 `optsDisplayName` 叙事 | 实际不再用于 alias | 简化 API 或改注释 |

| Rust 校验三份拷贝 | `is_valid_aliasget_template_by_alias` 内联校验、与 Go `validateTemplateAlias` 平行 | 统一函数并补 `tpl-snap-` |

| `image_job_runner` 里 `if info.Status != StatusFailed` | `finalizeTemplateReplicas` 失败时已 return | 可删或保留作防御，二选一写清 |

| 过时注释 | “newest wins” / “threading DisplayName via opts” / create 事务为 alias TOCTOU | 与 claim-after-READY 对齐 |

冗余本身不单独构成合入否决项，但 **2.1 的双轨路径**已经从“冗余”升级成“正确性脚枪”，应优先处理。

## 5. 合入标准核对

| 标准 | 是否满足 |

|------|----------|

| 核心功能可用且有 e2e | 是 |

| 先前 blocking（READY 前抢 alias / migration / v2 regression / 索引） | 基本是 |

| 文档宣称语义与实现一致 | 部分否（newest wins / claim soft-fail） |

| 旁路正确性（AgentHub delete-by-alias） | **否** |

| 无潜伏回归脚枪 | **否**（createDefinitionTx early claim） |

| reviewDecision / 人工 Approve | 否（仍为 CHANGES_REQUESTED） |

**合入建议**: 再修一轮后再合，最低要求：

1. 修 AgentHub reverse-sync canonical ID（§1.1）

2. 去掉或封死 `createDefinitionTx` 的 template early-claim（§2.1）

3. （强烈建议）claim 失败可观测 / 写回 DisplayName（§1.2）

修完后可视为达到合入标准；仅改措辞不改 1.1/2.1 **不建议**合入。
